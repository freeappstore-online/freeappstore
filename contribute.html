Become a Creator

Apply to join the creators program and ship your own app under the FreeAppStore platform.

What creators get

  • Your own app at yourapp.freeappstore.online
  • Auto-deploy on every push to main via GitHub Actions
  • Listed in the FreeAppStore catalog for all users to discover

How to apply

  1. Open an issue

    Go to github.com/freeappstore-online/submissions and describe your app idea. Include your GitHub username.

  2. We review within 48 hours

    We check the idea fits the platform — unique category, genuinely useful, buildable as a PWA.

  3. Get provisioned

    Once approved, we add you to the creators team, create your repo, set up your subdomain, and configure auto-deploy. You start coding.

Requirements

  • Free forever — no monetization, no ads, no paywalls
  • No tracking — zero analytics, cookies, or data collection
  • Open source — MIT license, public repo
  • Uses our tech stack — TypeScript, React, Vite, Tailwind. See Guidelines

Workspace Setup

Organize your repos under a consistent folder so every tool — and every AI agent — knows where to find things.

mkdir -p ~/dev/fas/apps && cd ~/dev/fas/apps

Clone each app repo into the apps/ folder:

git clone https://github.com/freeappstore-online/my-app.git # Result: ~/dev/fas/apps/my-app/

The recommended layout:

~/dev/fas/ apps/ ← your app repos timer/ notes/ calculator/ platform/ ← store site & templates (maintainers only)

This matches the paths in SKILLS.md, so AI agents reading the platform guide will find your code automatically. If you also build games, use ~/dev/fgs/games/ for FreeGameStore projects.

Start Building

The fas CLI handles everything end-to-end — scaffold, check, publish, list, logs.

  1. Install the CLI
    npm i -g @freeappstore/cli

    Requires Node 22+. Source on github.com/freeappstore-online/platform; package on npm.

  2. Sign in with GitHub
    fas login

    Device-flow OAuth — no password leaves your terminal. Token cached at ~/.fas/config.json (mode 0600).

  3. Scaffold a new app
    fas init my-app

    Clones the standalone template, replaces APPNAME placeholders, runs git init, makes the first commit. Use --template connected if your app needs the platform backend (auth + KV + rooms).

  4. Develop locally
    cd my-app && pnpm install && pnpm dev

    Edit web/src/App.tsx. The Shell component gives you the sidebar layout, brand fonts, and dark mode for free.

  5. Run compliance checks
    fas check

    Catches missing brand fonts, tracking SDKs, oversized bundles, missing PWA manifest, and unsubstituted template placeholders before you publish.

  6. Verify the layout fits (recommended)
    fas screencheck

    Builds, serves the dist locally, and drives a real Chromium at the min_viewport_width declared in your manifest in both portrait and landscape. Fails if the page actually scrolls — the static check only sees what you declared, this proves it works. Playwright is an opt-in peer dep; the command prompts to install on first run.

  7. Publish
    fas publish

    Provisions your repo at github.com/freeappstore-online/my-app, the hosting route, the custom subdomain, and the storefront entry. Prints the git remote add + git push commands when done.

    Scriptable from CI: pass --name, --category, --type, --oneliner, and --yes.

  8. Push your code
    git push upstream main

    Auto-deploys via GitHub Actions in ~30 seconds. Future commits to main deploy automatically too.

Other commands

  • fas list — see all apps you've published
  • fas logs my-app — tail deployment logs
  • fas doctor — health check (Node, git, pnpm, auth, API reachability)
  • fas whoami — show signed-in GitHub user

Full reference: github.com/freeappstore-online/platform/packages/cli

Brand and UI rules (mandated)

Every app on the platform looks predictable and consistent. The storefront, the detail pages, and every app share the same visual language so visitors learn the rhythm once and never relearn it. This is not a style guide suggestion — it's enforced by fas check and the CI workflow that runs before merge.

  • Use the template's CSS tokens. Colors, surfaces, lines, and spacing come from --paper, --ink, --muted, --accent, --line, --line-strong, --panel. Reference them via var(--accent), etc. Don't redefine them in component CSS or inline styles.
  • Brand fonts only: Manrope (body) and Fraunces (display). System and monospace fallbacks are fine. No custom display fonts (Cormorant, Sora, Inter, etc.) — even when they'd look great.
  • No custom theme. Light/dark mode comes from prefers-color-scheme and the template's variables. Don't ship your own theme switcher or override color-scheme.
  • Layout uses Shell. The sidebar layout, header, and footer rhythm come from web/src/components/Shell.tsx. Apps may add content inside Shell but shouldn't replace it.

The fas check compliance suite (run automatically before fas publish and again in your repo's CI) enforces:

  • No template placeholders — every APPNAME substituted
  • No tracking SDKs — none of the 8 known trackers (Google Analytics, Plausible, Mixpanel, etc.)
  • Brand fonts present — Manrope and Fraunces both referenced
  • No brand overrides — no app redefines a platform CSS token outside the canonical theme file; no non-brand font-family declarations
  • PWA manifest — web/public/manifest.json with name, display, start_url
  • Bundle size — main bundle under 300 KB gzipped

Run them locally any time:

fas check

Hard failures block fas publish (override with --skip-checks at your own risk; admin review will still catch issues). Warnings are advisory.
