MCP Server

Build, publish, and improve real FreeAppStore apps without leaving your editor. Connect once, then ask Cursor or Claude to "build me a habit tracker and deploy it" — the app goes live at a .freeappstore.online subdomain.

What is MCP?

Model Context Protocol (MCP) is an open standard that lets AI assistants call tools on remote servers. Once you connect FreeAppStore as an MCP server, your editor's AI can do real work on your behalf — provision an app, write and deploy its code, read logs, check deploys, and look up the SDK — all as tools it calls directly, no copy-pasting or context-switching.

Two ways to build an app

Both happen entirely from your editor. Pick whichever fits how you like to work:

ModelWho writes the codeToolsAI key needed?
Your AI writes the code Your editor's model (Claude / Cursor) writes the code, and the MCP ships it to FreeAppStore. create_app, update_files, read_file, list_files No — you're already paying your editor's model.
The platform's agent writes the code You send a natural-language prompt; FreeAppStore's own VibeCode agent writes and deploys the app for you. agent_build, agent_status Yes — a funded AI key stored in your platform key vault.

Connect

The server speaks Streamable HTTP at https://mcp.freeappstore.online/mcp.

Claude Code

One command:

claude mcp add freeappstore -- npx mcp-remote https://mcp.freeappstore.online/mcp

Or add it to ~/.claude.json by hand:

{
  "mcpServers": {
    "freeappstore": {
      "command": "npx",
      "args": ["mcp-remote", "https://mcp.freeappstore.online/mcp"]
    }
  }
}

Cursor

Settings → MCP → Add server:

Name: FreeAppStore
Command: npx mcp-remote https://mcp.freeappstore.online/mcp

Any MCP-compatible client

Point it at https://mcp.freeappstore.online/mcp (Streamable HTTP transport). Discovery metadata lives at https://freeappstore.online/.well-known/mcp.json.

Authentication

Read-only reference tools (platform_guide, sdk_reference) need no auth. Any tool that reads or writes your apps — building, deploying, listing, reading logs — needs a FreeAppStore session token passed as Authorization: Bearer <token>.

Get a token with the CLI:

npm i -g @freeappstore/cli
fas login

fas login completes the GitHub sign-in and stores the token in ~/.fas/config.json under session.token. To pass it through mcp-remote, append a header argument to the args. For example, in ~/.claude.json:

{
  "mcpServers": {
    "freeappstore": {
      "command": "npx",
      "args": [
        "mcp-remote",
        "https://mcp.freeappstore.online/mcp",
        "--header",
        "Authorization: Bearer <your-session-token>"
      ]
    }
  }
}

Tool reference

Build — your AI authors the code

Your editor's model writes the files; these tools provision and ship them. Writes require a FreeAppStore session and owner access.

ToolAuthDescription
create_app(app_id, category, oneliner, type?, description?) FAS session Provision the GitHub repo + R2 hosting + store listing, scaffold the starter template, and deploy. The app goes live at <app_id>.freeappstore.online. type is "standalone" (default) or "connected" (wires in the @freeappstore/sdk).
update_files(app_id, files:[{path, content}], message?) FAS session (owner) Write or overwrite files in your app's repo. Auto-redeploys in ~30–60s.
read_file(app_id, path) FAS session Read a single file from an app's repo.
list_files(app_id) FAS session List the files in an app's repo.

Agent — the platform's VibeCode AI authors the code

Hand off a prompt and let FreeAppStore's agent do the building. Needs a funded AI key in your platform key vault matching the chosen provider.

ToolAuthDescription
agent_build(prompt, provider?, model?, session_id?) FAS session + AI key Hand a natural-language prompt to the VibeCode agent; it writes the code and deploys it. provider is anthropic (default) / openai / openrouter / google and must match a key in your vault. Long-running — returns a session_id to poll.
agent_status(session_id) FAS session Poll a build session: app id, deploy phase, and live URL.

Inspect — see what you've shipped

ToolAuthDescription
list_apps() FAS session List your published apps with categories, descriptions, and URLs.
app_info(app_id) FAS session Live URL, repo, store listing, and current status for an app.
deploy_status(app_id) FAS session Recent GitHub Actions deploy runs — pass/fail, commit SHA, timestamp.
app_logs(app_id, level?, limit?) FAS session (owner) Recent errors, warnings, SDK calls, and build info for debugging.

Reference — no auth

ToolAuthDescription
platform_guide() None Returns the full SKILLS.md — the complete platform guide (tech stack, project structure, SDK, CLI, compliance, deploy flow).
sdk_reference(feature?) None Quick @freeappstore/sdk reference. Filter by feature: auth, kv, counters, collections, rooms, proxy, keys, hooks, ui.

Example session

Once connected, a build is just a conversation. Two flavors:

Let the platform agent do everything:

You:  Build me a habit tracker called daily-streak and deploy it.
AI:   [calls agent_build with prompt + provider "anthropic"]
      → returns session_id "sess_9f2a…"
You:  Is it done yet?
AI:   [calls agent_status("sess_9f2a…")]
      → phase: "deployed", live at https://daily-streak.freeappstore.online

Or have your own editor write the code:

You:  Create a connected app daily-streak in the "productivity" category,
      then build out the habit-tracking UI.
AI:   [create_app("daily-streak", "productivity", "Track daily habits", type "connected")]
      [list_files / read_file to inspect the scaffold]
      [update_files(...) with the screens and SDK wiring]
You:  Did the deploy pass?
AI:   [deploy_status("daily-streak")] → latest run: success
      → live at https://daily-streak.freeappstore.online
